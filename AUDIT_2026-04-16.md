## Section 1 тАФ System.out / System.err Usage

Four call-sites found. None use `System.err`.

- **`src/main/java/frc/robot/Robot.java:106`** тАФ `System.out.printf("[Robot] %s тЖТ %s (phase lasted %.2fs)%n", ...)` inside `logPhaseTransition()`. Called on every mode transition (4 times per match at most). Not a loop-rate call. Transition data is already logged via `Logger.recordOutput` on lines 103тАУ105. **Recommendation:** remove `System.out.printf`; the three preceding `Logger.recordOutput` lines already capture the same information.

- **`src/main/java/frc/robot/Helper.java:89`** тАФ `System.out.println(rpm)` inside `printRpmDistance()`. This method is called every 20 ms cycle from `FlywheelStatic.execute()`, `FlywheelAutoFeed.execute()`, and `FlywheelAim.execute()` via `if (printCounter % 10 == 0)` gate тАФ effective rate 5 Hz. **CRITICAL: loop-rate console I/O at 5 Hz will saturate the RoboRIO serial buffer and can contribute to CAN frame drops and watchdog overruns during a match.** Recommendation: replace `printRpmDistance` call-sites with `Logger.recordOutput("Flywheel/DebugRpm", rpm)` gated the same way, and delete `printRpmDistance`.

- **`src/main/java/frc/robot/Helper.java:90`** тАФ `System.out.println(distance)` тАФ same function, same loop-rate issue as above. See finding above.

- **`src/main/java/frc/robot/Helper.java:91`** тАФ `System.out.println("*****************")` тАФ same function, same loop-rate issue as above. Decorative separator with no diagnostic value. Remove.

## Section 3 тАФ Brownout Scale Consumption

`getBrownoutScale()` is defined at `src/main/java/frc/robot/Robot.java:94`.

- `DriveToGamePieceCommand.execute()` тАФ speed cap is `Math.min(nearestDist * kTranslationP, 1.0)` times `kMaxSpeedMetersPerSec` (line 75), no brownout scale. Recommendation: apply scale.
